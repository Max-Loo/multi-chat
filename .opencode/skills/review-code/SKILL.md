# 代码审查 Skill

审查变更的代码，识别与复用性、质量和效率相关的问题。

## 触发场景

- 用户说 "review code" 或 "审查代码"
- 用户说 "检查代码质量"
- 用户说 "code review"
- 用户想要对当前变更进行代码审查

## 输入参数

`/review-code` 可选接受：
- **无参数**：审查所有未提交的变更（已暂存 + 未暂存）
- **文件路径**：审查指定文件（例如：`/review-code src/utils.ts`）
- **目录**：审查目录中所有变更的文件

## 工作流程

### 1. 识别变更代码

运行以下 git 命令识别所有变更的代码：

```bash
git diff
git diff --cached
```

如果提供了特定文件或目录，则聚焦于这些文件。

### 2. 分析代码质量

从以下维度审查变更的代码：

**代码复用**
- 可提取的重复代码块
- 可泛化的相似模式
- 可利用的现有工具函数/函数

**代码质量**
- 不清晰或误导性的变量/函数名
- 缺失或错误的类型定义
- 不可达或死代码
- 可简化的过于复杂的逻辑
- 缺失边界情况的错误处理

**效率**
- 不必要的重新渲染（React 组件）
- 内存泄漏（未取消订阅的监听器、未清除的定时器）
- 低效的算法或数据结构
- 缺失的缓存优化机会
- 不必要的计算或 API 调用

**最佳实践**
- 违反项目约定（见 AGENTS.md）
- 安全漏洞（XSS、注入等）
- 可访问性问题
- 缺失或不足的测试

### 3. 输出结果

**如果发现问题**，以结构化格式呈现：

```markdown
## 🔍 代码审查结果

### 概述
| 类别     | 发现问题数 |
|----------|-----------|
| 复用性   | X         |
| 质量     | Y         |
| 效率     | Z         |

### 问题列表

#### 1. [类别] 问题描述
**文件**：`path/to/file.ts:123`
**问题**：问题的描述
**建议**：如何修复
**优先级**：高/中/低

#### 2. ...
```

**如果未发现问题**：

```markdown
✅ 未发现重大问题。代码看起来不错！
```

## 审查原则

- **建设性**：提供可操作的建议，而不仅仅是批评
- **具体**：引用确切的文件路径和行号
- **优先级**：区分关键问题和次要建议
- **上下文**：考虑项目约定和现有模式
- **非侵入性**：仅报告，不自动修复

## Guardrails

- **不要修改任何文件**：这是只读审查
- **不要自动修复问题**：仅报告发现
- **不要提交变更**：不执行 git 操作
- **务实**：关注有意义的问题，而非吹毛求疵
