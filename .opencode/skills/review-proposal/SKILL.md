# 方案审查 Skill

深度审查模式：仔细阅读并分析用户提供的方案，寻找潜在问题。

## 输入格式

用户可通过以下方式提供审查对象：

- **文件路径**：如 `./docs/design.md` - 读取指定文件
- **URL**：如 `https://example.com/proposal.md` - 获取远程文档
- **目录路径**：列出目录文件，读取相关文档（如 .md、.txt）
- **方案文本**：直接粘贴的方案内容
- **空白输入**：使用 AskUserQuestion 工具询问审查对象

## 审查流程

### 1. 确定审查对象

根据输入类型处理：

| 输入类型 | 处理方式                              |
| -------- | ------------------------------------- |
| 文件路径 | 读取指定文件内容                      |
| URL      | 使用 WebFetch 工具获取内容            |
| 目录路径 | 列出目录文件，读取相关文档            |
| 方案文本 | 直接使用提供的文本                    |
| 无输入   | 使用 AskUserQuestion 工具询问审查对象 |

### 2. 深度分析

从以下维度进行审查：

**逻辑一致性**

- 前后是否存在矛盾
- 假设是否合理
- 推理链条是否完整

**技术可行性**

- 技术选型是否合理
- 实现路径是否清晰
- 是否遗漏关键技术点

**架构设计**

- 组件职责是否清晰
- 依赖关系是否合理
- 是否存在循环依赖

**性能与可维护性**

- 性能瓶颈是否考虑
- 代码可维护性
- 测试覆盖是否充分

**安全与风险**

- 安全隐患
- 潜在风险点
- 错误处理是否完善

**优化机会**

- 是否有更优实现方案
- 是否可以简化
- 是否有可复用的现有方案

### 3. 输出结果

#### 发现问题时

以结构化方式列出：

```markdown
## 🚨 发现的问题

### 1. [问题类型] - 简要描述

**位置**：具体位置引用（如文件路径、章节名）
**问题**：详细描述问题
**建议**：改进方案
**优先级**：高/中/低

### 2. ...
```

#### 未发现明显问题时

简单肯定方案质量：

> ✅ 方案整体清晰合理，逻辑自洽，可以开始实施。

## 审查原则

- **建设性**：指出问题的同时提供改进建议
- **具体性**：引用具体位置，避免泛泛而谈
- **优先级**：区分致命问题和优化建议
- **尊重原意**：不改变方案核心目标，只优化实现路径
- **可执行性**：建议应该是可操作的，而非理论讨论

## Guardrails

- **不要修改原文件**：只输出审查结果
- **不要自动修复**：指出问题，但不直接修改
- **保持客观**：基于技术标准，而非个人偏好
- **适度审查**：避免过度优化或吹毛求疵
