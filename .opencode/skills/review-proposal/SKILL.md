# 方案审查 Skill

深度审查模式：仔细阅读并分析用户提供的方案，寻找潜在问题。

## 输入格式

用户可通过以下方式提供审查对象：

- **文件路径**：如 `./docs/design.md` - 读取指定文件
- **URL**：如 `https://example.com/proposal.md` - 获取远程文档
- **目录路径**：列出目录文件，读取相关文档（如 .md、.txt）
- **方案文本**：直接粘贴的方案内容
- **空白输入**：使用 AskUserQuestion 工具询问审查对象

## 审查流程

### 1. 确定审查对象

根据输入类型处理：

| 输入类型 | 处理方式                              |
| -------- | ------------------------------------- |
| 文件路径 | 读取指定文件内容                      |
| URL      | 使用 WebFetch 工具获取内容            |
| 目录路径 | 列出目录文件，读取相关文档            |
| 方案文本 | 直接使用提供的文本                    |
| 无输入   | 使用 AskUserQuestion 工具询问审查对象 |

### 2. 深度分析

从以下维度进行审查：

**逻辑一致性**

- 前后是否存在矛盾
- 假设是否合理
- 推理链条是否完整

**技术可行性**

- 技术选型是否合理
- 实现路径是否清晰
- 是否遗漏关键技术点

**架构设计**

- 组件职责是否清晰
- 依赖关系是否合理
- 是否存在循环依赖

**性能与可维护性**

- 性能瓶颈是否考虑
- 代码可维护性
- 测试覆盖是否充分

**安全与风险**

- 安全隐患
- 潜在风险点
- 错误处理是否完善

**优化机会**

- 是否有更优实现方案
- 是否可以简化
- 是否有可复用的现有方案

### 3. 输出结果

#### 发现问题时

以结构化方式列出，**必须按优先级从高到低排序**：

```markdown
## 🚨 发现的问题

> **优先级定义**：
> - 🔴 **P0 - 阻断**：方案存在根本性缺陷，不修复则无法实施
> - 🟠 **P1 - 严重**：会导致功能异常、数据丢失或安全漏洞，必须在实施前修复
> - 🟡 **P2 - 中等**：影响可维护性、性能或用户体验，建议在实施中修复
> - 🟢 **P3 - 建议**：优化改进项，可择机处理

---

### 1. [问题类型] - 简要描述 🔴 P0

**位置**：具体位置引用（如文件路径、章节名）
**问题**：详细描述问题
**建议**：改进方案
**优先级判定原因**：说明为什么判定为此优先级（如：该问题会导致核心功能无法运行 / 该假设不成立将使整个方案失效）

### 2. ...
```

每个问题**必须包含「优先级判定原因」**，解释为何归为该级别而非相邻级别。判定依据应具体、客观，例如：

- P0 判定依据：方案的核心假设不成立、关键逻辑自相矛盾、遗漏必要依赖
- P1 判定依据：会导致运行时错误、数据不一致、安全漏洞、无法回滚
- P2 判定依据：增加维护成本、存在性能瓶颈、违反已有项目约定
- P3 判定依据：存在更优实现方式、可简化但当前方案可行、风格改进

#### 未发现明显问题时

简单肯定方案质量：

> ✅ 方案整体清晰合理，逻辑自洽，可以开始实施。

## 审查原则

- **建设性**：指出问题的同时提供改进建议
- **具体性**：引用具体位置，避免泛泛而谈
- **优先级分级**：使用 P0-P3 四级优先级，每个问题必须说明判定原因
- **尊重原意**：不改变方案核心目标，只优化实现路径
- **可执行性**：建议应该是可操作的，而非理论讨论

## Guardrails

- **不要修改原文件**：只输出审查结果
- **不要自动修复**：指出问题，但不直接修改
- **保持客观**：基于技术标准，而非个人偏好
- **适度审查**：避免过度优化或吹毛求疵
