# i18n-ui-text

## Purpose

提供完整的 UI 文本国际化能力，支持应用在简体中文、英文、法语之间无缝切换。确保所有用户可见的界面文本都通过 i18next 配置提供，避免硬编码文本，提升应用的国际化能力和用户体验。

## Requirements

### Requirement: 用户界面文本必须使用国际化配置

所有用户可见的界面文本（包括按钮、标签、占位符、错误提示、标题等）必须通过 i18next 配置提供，不得在代码中硬编码中英文字符串。

**适用范围**：
- React 组件中渲染到 DOM 的文本内容
- HTML 属性中的用户可见文本（如 `title`、`aria-label`、`alt` 等）
- 用户交互相关的提示信息
- 表单验证错误消息

**排除范围**：
- 测试代码中的文本（`src/__test__/` 目录）
- 代码注释和文档字符串
- 控制台日志和警告信息（`console.log`、`console.warn`、`console.error`）
- 开发者调试信息
- 技术标识符（如 HTML `id`、CSS `class`、数据结构键名）

#### Scenario: 组件中硬编码的中文按钮文本

- **WHEN** 组件包含硬编码的中文文本（如 `title="新建聊天"`）
- **THEN** 系统必须将该文本提取到对应的命名空间（如 `navigation.json`）并使用 `useTranslation()` hook 替换为 `t('navigation.newChat')`

#### Scenario: 组件中硬编码的英文 ARIA 标签

- **WHEN** 组件包含硬编码的英文文本（如 `aria-label="Loading"`）
- **THEN** 系统必须将该文本提取到对应的命名空间并使用 `t()` 函数替换为国际化键值

#### Scenario: 占位符文本国际化

- **WHEN** 输入框包含硬编码的占位符文本（如 `placeholder="搜索聊天..."`）
- **THEN** 系统必须将该占位符文本提取到国际化配置并使用 `t()` 函数替换

### Requirement: 国际化键值必须遵循命名空间约定

国际化键值必须根据其所属功能模块放置在正确的命名空间文件中，保持清晰的组织结构。

**命名空间映射**：
- `common.json`：通用 UI 文本（按钮、状态、常见操作）
- `navigation.json`：导航相关文本（菜单、标签页、面包屑）
- `chat.json`：聊天功能相关文本（消息、发送、附件）
- `model.json`：模型管理相关文本（模型名称、参数）
- `setting.json`：设置页面相关文本（配置选项、表单）
- `table.json`：表格组件相关文本（列标题、操作按钮）
- `provider.json`：供应商相关文本（供应商名称、配置）

**键值命名约定**：
- 最大嵌套层数：3 层（如 `<namespace>.<feature>.<element>`）
- 示例：`navigation.chat.newChat`（✅ 3 层）
- 反例：`navigation.chat.sidebar.actions.newChat`（❌ 4 层）

#### Scenario: 按钮文本命名空间分配

- **WHEN** 添加新的按钮文本翻译
- **THEN** 必须根据按钮的功能和位置选择合适的命名空间（如"保存"按钮 → `common.submit`，"新建聊天"按钮 → `navigation.newChat`）

#### Scenario: 新增命名空间键值

- **WHEN** 现有命名空间无法准确分类新的翻译文本
- **THEN** 必须评估是否需要创建新的命名空间文件，或扩展现有命名空间的结构

### Requirement: 国际化键值必须遵循冲突解决机制

国际化键值必须遵循决策树机制，避免命名冲突和重复定义。

**键值冲突解决决策树**：
```
优先级 1：是否跨组件复用？
  → 是 → 使用 common.*
  → 否 → 继续判断

优先级 2：是否页面级导航？
  → 是 → 使用 navigation.*
  → 否 → 继续判断

优先级 3：是否特定功能？
  → 是 → 使用 {chat|model|setting|table|provider}.*
  → 否 → 评估是否需要新建命名空间
```

**跨命名空间引用**：
- 支持使用 i18next 的引用功能（如 `common:@:common.submit`）
- 避免在多个命名空间中重复定义相同的文本

#### Scenario: 跨组件复用的文本

- **WHEN** 文本在多个组件中复用（如"保存"、"取消"、"删除"）
- **THEN** 必须使用 `common.*` 命名空间，避免在每个功能命名空间中重复定义

#### Scenario: 深层嵌套的键值

- **WHEN** 键值嵌套层数超过 3 层（如 `navigation.chat.sidebar.actions.newChat`）
- **THEN** 必须简化为 3 层或更少（如 `navigation.chat.newChat`）或创建新的命名空间

### Requirement: 所有语言必须保持翻译完整性

所有支持的语言（简体中文 `zh`、英文 `en`、法语 `fr`）必须提供完整的翻译，确保每个键值在所有语言文件中都存在。

**完整性要求**：
- 以英文翻译为基准，确保所有语言包含相同的键值结构
- 支持嵌套键值的递归完整性检查
- 使用自动化工具（`scripts/check-i18n.js`）验证翻译完整性

#### Scenario: 添加新的翻译键值

- **WHEN** 在一个语言文件中添加新的翻译键值
- **THEN** 必须同时在所有其他语言文件中添加对应的翻译（在 `zh/`、`en/`、`fr/` 的对应命名空间中添加相同键值）

#### Scenario: 验证翻译完整性

- **WHEN** 完成国际化修改后
- **THEN** 必须运行 `npm run lint:i18n` 验证所有语言的翻译完整性，确保没有遗漏的键值

### Requirement: 国际化替换必须保持原有功能

将硬编码文本替换为国际化调用时，必须确保不改变原有的功能逻辑、样式和用户体验。

**兼容性要求**：
- 保持组件的 props 和 API 不变
- 保持动态文本插值功能（如 `t('key', { value })`）
- 保持文本的格式和样式（换行、加粗、链接等）

#### Scenario: 动态文本插值

- **WHEN** 硬编码文本包含动态变量（如 `已选择 ${count} 个项目`）
- **THEN** 国际化替换必须支持变量插值（如 `t('common.selectedCount', { count })`）且翻译文件中包含 `{{count}}` 占位符

#### Scenario: 格式化文本保留

- **WHEN** 硬编码文本包含格式化（如换行符 `\n`、HTML 标签）
- **THEN** 国际化配置必须保留相同的格式，或在组件中使用 `Trans` 组件处理复杂格式

### Requirement: 控制台和日志信息不受国际化约束

开发者调试信息、控制台日志和警告信息不需要国际化，可以保持硬编码英文或中文。

**允许硬编码的场景**：
- `console.log()`、`console.warn()`、`console.error()` 中的文本
- 开发者工具中的调试输出
- 错误堆栈信息
- 技术性能监控日志

#### Scenario: 控制台警告信息

- **WHEN** 代码中包含控制台警告（如 `console.warn('获取系统语言失败，使用英文', error)`）
- **THEN** 该警告信息可以保持硬编码，不需要提取到国际化配置

#### Scenario: 错误日志记录

- **WHEN** 记录错误日志用于调试
- **THEN** 错误消息可以保持硬编码英文，便于开发者理解和搜索问题

### Requirement: 动态内容必须支持复数和上下文相关翻译

包含动态变量的文本必须支持 i18next 的复数、上下文和插值功能，确保不同语言的正确表达。

**复数处理**：
- 使用 i18next 的复数功能：`t('item', { count })`
- 翻译文件中必须包含单数和复数形式（使用 `_plural` 后缀或语言特定的复数规则）

**上下文相关翻译**：
- 使用 i18next 的 context 功能区分相同单词在不同上下文的含义
- 示例：`confirm_dialog`（对话框中的"确认"）vs `confirm_action`（操作的"确认"）

**动态实体名称**：
- 预定义列表（如模型名称）：在翻译文件中添加所有可能值
- 用户自定义（如聊天标题）：保持原样，不翻译
- 动态构建：使用 i18next 插值功能（`{{variable}}`）

#### Scenario: 复数形式的翻译

- **WHEN** 文本包含数量词（如"1 个项目" vs "2 个项目"）
- **THEN** 必须使用 i18next 的复数功能（如 `t('item', { count })`）且翻译文件中包含单数和复数形式

#### Scenario: 上下文相关的翻译

- **WHEN** 相同单词在不同上下文有不同含义（如"确认"按钮 vs "确认"对话框标题）
- **THEN** 必须使用 i18next 的 context 功能（如 `confirm_dialog` vs `confirm_action`）区分不同上下文

#### Scenario: 动态实体名称的处理

- **WHEN** 文本包含动态实体名称（如模型名称、供应商名称）
- **THEN** 如果是预定义列表，在翻译文件中添加所有可能值；如果是用户自定义，保持原样不翻译
