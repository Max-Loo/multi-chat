# 设计文档：AGENTS.md 重构

## Context

### 当前状态

AGENTS.md 当前包含 447 行内容，结构如下：
- **文档维护原则**（56 行）：内容分类规则、检查清单、维护规范
- **项目概述**（9 行）：技术栈和架构简介
- **架构**（14 行）：前后端架构
- **开发命令**（30 行）：常用命令和测试规范
- **关键设计说明**（140 行）：7 个设计主题的详细说明
- **开发规范**（82 行）：导入路径、代码文档、实现原则等
- **项目约定**（32 行）：时间戳工具、Tauri 命令添加
- **文档参考**（72 行）：按功能和架构层次的文件索引表
- **文件结构**（6 行）：顶层目录结构

### 问题分析

1. **信息密度不均**：核心指导信息与详细设计说明混在一起
2. **AI 读取效率低**：大量详细信息干扰快速定位核心概念
3. **可维护性差**：设计说明更新时需要修改大文件
4. **缺乏层次**：从查找到深度探索的内容都在一个文件中

### 约束条件

- **零代码变更**：本次重构仅涉及文档，不影响代码实现
- **向后兼容**：所有现有信息必须保留（只是重新组织）
- **长期可维护**：建立清晰的规范，防止文档再次膨胀
- **AI 友好**：AGENTS.md 精简后便于 AI 快速读取核心信息

## Goals / Non-Goals

### Goals

- **精简 AGENTS.md**：从 447 行减少到约 150 行（减少 66%）
- **建立文档层次**：创建 docs/design/ 和 docs/conventions/ 子目录
- **嵌入维护规范**：将文档拆分决策树写入 AGENTS.md 的"文档维护原则"
- **完整保留信息**：所有现有内容迁移到合适的位置
- **建立索引体系**：docs/README.md 提供完整的文档导航

### Non-Goals

- **不修改代码**：代码实现保持不变
- **不删除信息**：所有现有内容都迁移到合适位置
- **不改变格式**：Markdown 格式和写作风格保持一致
- **不引入新工具**：使用现有的文档工具（无需构建流程）

## Decisions

### 决策 1：AGENTS.md 保留内容

**选择**：仅保留核心指导信息，删除详细说明和示例

**理由**：
- AGENTS.md 是 AI 首先读取的文档，应该快速传达核心概念
- 详细说明更适合放在专门的子文档中
- 减少行数可以提高可读性和维护性

**保留内容**：
1. 文档维护原则（~120 行，扩展维护规范）
2. 项目概述（~9 行）
3. 架构概览（~14 行）
4. 开发命令（~20 行，简化）
5. 开发规范（~50 行）
6. 设计文档索引（~15 行，新增）
7. 项目约定索引（~10 行，新增）
8. 快速查找表（~20 行，简化）
9. 文件结构（~5 行，简化）

**删除/迁移内容**：
- 详细设计说明 → docs/design/
- 项目约定详情 → docs/conventions/
- 完整文件索引表 → docs/README.md
- 代码示例 → 保留简要说明，详细内容指向源文件

- 易于阅读和输入

**示例**：

- ✅ `lazy-loading.md`

**新增内容**：

3. **docs/ 目录结构规范**：明确各子目录的用途
4. **维护规范**：目标行数从 350 调整到 150

## Risks / Trade-offs

### 风险 1：文档链接失效

**风险**：迁移后，外部引用或内部链接可能失效

**缓解措施**：
- 使用相对路径链接（如 `[初始化流程](docs/design/initialization.md)`）
- 在迁移完成后运行链接检查工具（如 `markdown-link-check`）
- 在 AGENTS.md 中保留"快速查找表"，提供核心文件路径

### 风险 2：AI 未读取子文档

**风险**：AI 可能只读取 AGENTS.md，错过详细的子文档

**缓解措施**：
- 在 AGENTS.md 中明确指出详细信息在子文档中
- 使用清晰的索引和一句话概述
- 在"文档维护原则"中强调子文档的重要性

### 风险 3：维护成本增加

**风险**：分散的文档可能增加维护成本

**缓解措施**：
- 建立清晰的文档同步要求（见 design.md 的"文档同步要求"）
- 每季度审查文档行数和内容质量
- 使用"文档拆分决策树"减少决策成本

### 风险 4：子文档内容过时

**风险**：子文档可能长期未更新，导致内容过时

**缓解措施**：
- 在代码注释中保留关键信息（优先级最高）
- 在 AGENTS.md 的"文档同步要求"中明确更新责任
- 每季度审查时检查子文档的时效性

### 权衡：文档粒度

**权衡**：子文档的数量和粒度

**选择**：
- 设计文档按主题拆分（6 个文档）
- 约定文档按主题拆分（2 个文档）
- 避免过度拆分（如每个功能一个文档）

**理由**：
- 平衡可维护性和可发现性
- 太多文档会增加导航成本
- 太少文档会失去拆分的意义
