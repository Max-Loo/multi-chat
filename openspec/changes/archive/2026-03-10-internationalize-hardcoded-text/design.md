# UI 文本国际化技术设计

## Context

**当前状态**：
- 项目已建立 i18next + react-i18next 基础设施
- 支持三种语言：简体中文（zh）、英文（en）、法语（fr）
- 实现了英文"第一公民"按需加载策略和翻译完整性检查工具
- **部分组件已使用国际化**（如 `common.json`、`navigation.json` 等），但仍有 2000+ 处硬编码文本需要处理

**问题**：
- 前端代码中存在 2000+ 处硬编码的中英文文本
- 用户无法切换语言体验不同语言界面
- 新增语言支持成本高，需要逐个修改组件

**约束条件**：
- 必须保持现有功能不变，避免回归
- 不能破坏现有的英文按需加载策略
- 必须通过翻译完整性检查
- 不能影响测试代码（测试文件中的硬编码文本需要保留）

**相关方**：
- 前端开发团队：负责实施国际化替换
- 用户体验团队：提供翻译内容
- 质量保证团队：验证多语言功能

## Goals / Non-Goals

**Goals**：
1. 识别并提取所有用户可见的硬编码文本到国际化配置
2. 替换组件中的硬编码文本为 `useTranslation()` 或 `t()` 调用
3. 确保所有支持语言的翻译完整性
4. 建立清晰的命名空间组织结构，便于维护和扩展

**Non-Goals**：
1. 不处理测试代码中的硬编码文本（`src/__test__/` 目录保持不变）
2. 不国际化控制台日志和开发者调试信息
3. 不改变现有的 i18next 配置和加载策略
4. 不引入新的国际化库或工具链

## Decisions

### 决策 1：使用现有的 i18next + react-i18next 基础设施

**选择**：继续使用项目已配置的 i18next 和 react-i18next，而非引入新的国际化方案。

**理由**：
- 项目已有完善的 i18next 配置和按需加载策略
- 团队熟悉该技术栈，学习成本低
- 无需重构现有代码，降低风险

**替代方案**：
- **格式化消息（Format.js）**：功能更强大，但需要引入新依赖和重构
- **自定义国际化方案**：灵活性高，但维护成本大，重复造轮子

### 决策 2：按命名空间模块化组织翻译文件

**选择**：根据功能模块将翻译键值分配到 7 个命名空间（common、navigation、chat、model、setting、table、provider）。

**理由**：
- 清晰的职责划分，便于查找和维护
- 符合项目现有的文件组织结构
- 支持按需加载，减少初始 bundle 大小

**键值命名约定**：
```
<namespace>.<feature>.<element>
示例：navigation.chat.newChat（聊天导航中的新建聊天按钮）
最大嵌套层数：3 层
```

**键值冲突解决机制（决策树）**：
```
优先级 1：是否跨组件复用？
  → 是 → common.*（如"保存"、"取消"）
  → 否 → 继续判断

优先级 2：是否页面级导航？
  → 是 → navigation.*（如"新建聊天"、"返回"）
  → 否 → 继续判断

优先级 3：是否特定功能？
  → 是 → {chat|model|setting|table|provider}.*
  → 否 → 评估是否需要新建命名空间
```

**跨命名空间引用**（i18next 支持）：
```json
// navigation.json
{
  "save": "common:@:common.submit"  // 引用 common.submit
}
```

**替代方案**：
- **单一扁平文件**：简单但难以维护，不符合现有结构
- **按页面划分**：粒度过细，导致键值重复和冗余

### 决策 3：使用两阶段识别硬编码文本

**选择**：第一阶段使用粗略搜索（grep/ripgrep），第二阶段使用 AST 静态分析精确识别。

**理由**：
- **第一阶段（粗略搜索）**：快速发现潜在硬编码文本，覆盖范围广
- **第二阶段（AST 分析）**：准确识别 JSX 文本节点和 HTML 属性值，自动排除注释和技术标识符
- 误报率从 30-50% 降低到 5% 以下
- 可重复执行，便于验证完整性

**第一阶段：粗略搜索模式**：
```bash
# 搜索中文字符：[\u4e00-\u9fa5]+
# 搜索常见英文 UI 词汇：(Save|Delete|Add|Edit|Cancel|Loading|Error...)
# 搜索 HTML 属性中的文本：title、aria-label、placeholder、alt
```

**第二阶段：AST 静态分析**（使用 @babel/parser）：
- 准确识别 JSX 文本节点：`<Button>保存</Button>`
- 准确识别 JSX 属性值：`<Button title="新建聊天" />`
- 自动排除注释：`// 保存用户数据`
- 自动排除技术标识符：`const ERROR_CODE = 'ERROR_FAILED'`
- 识别模板字面量：`` `Failed: ${error}` ``

**人工审查过滤**：
- 排除开发者调试信息
- 排除构建时的字符串拼接
- 确认误报并调整 AST 分析规则

**替代方案**：
- **纯手工查找**：易遗漏，效率低
- **仅使用正则表达式**：误报率高（30-50%），人工成本大
- **仅使用 AST 分析**：初期投入大，但长期更准确

### 决策 4：按页面完成的渐进式替换策略

**选择**：按页面完成国际化（每个阶段结束时，至少有完整的用户路径可用），而非按命名空间分割。

**理由**：
- **避免混合状态**：防止同一页面出现"部分国际化、部分硬编码"的混乱体验
- 降低风险，便于测试和验证
- 可以并行开发，提高效率
- 每个阶段都有可交付的用户价值

**实施顺序（按页面完成）**：
```
阶段 1：导航和通用组件（所有页面共享）
  - 完成 common.json 和 navigation.json
  - 覆盖范围：通用按钮、状态、导航菜单
  - 验证：所有页面的导航和通用按钮已国际化

阶段 2：聊天页面（100% 国际化）
  - 完成 chat.json 及聊天相关的所有命名空间
  - 覆盖范围：聊天列表、聊天内容、发送框、附件
  - 验证：聊天页面完整可用，无硬编码残留

阶段 3：模型管理页面（100% 国际化）
  - 完成 model.json 及模型管理相关的所有命名空间
  - 覆盖范围：模型列表、模型配置、供应商管理
  - 验证：模型管理页面完整可用

阶段 4：设置页面（100% 国际化）
  - 完成 setting.json 及设置相关的所有命名空间
  - 覆盖范围：语言设置、主题设置、账户设置
  - 验证：设置页面完整可用
```

**混合状态的处理**：
- 如果必须分阶段（如时间紧迫），确保：
  - 同一页面内不出现混合状态
  - 至少有完整的用户路径可用
  - 在测试环境验证用户体验可接受

**替代方案**：
- **按命名空间优先级**：导致混合状态，体验混乱
- **一次性全量替换**：风险高，冲突多，难以调试

### 决策 5：保留控制台日志的硬编码文本

**选择**：不国际化 `console.log`、`console.warn`、`console.error` 中的文本。

**理由**：
- 控制台日志面向开发者，不是最终用户
- 英文技术日志便于全球开发者协作和问题搜索
- 减少不必要的翻译工作，聚焦用户体验

**允许硬编码的场景**：
```typescript
console.warn('获取系统语言失败，使用英文', error); // ✅ 允许
console.error(`Failed to load language ${lang}:`, error); // ✅ 允许
```

**需要国际化的场景**：
```typescript
toast.error('操作失败'); // ❌ 需要国际化
alert('保存成功'); // ❌ 需要国际化
```

### 决策 6：动态内容的国际化策略

**选择**：根据动态内容的类型使用不同的国际化策略。

**动态实体名称**（如模型名称、供应商名称）：
- **预定义列表**（如 GPT-4、Claude 3.5 Sonnet）：在翻译文件中添加所有可能值
- **用户自定义**（如聊天标题）：保持原样，不翻译
- **动态构建**（如 `${modelName} 模型`）：使用 i18next 插值功能

**复数处理**：
- 使用 i18next 的复数功能：`t('item', { count })`
- 翻译文件中使用 `{{count}}` 和后缀（`_0`、`_1`、`_2`）
```json
{
  "item": "one item",
  "item_plural": "{{count}} items"
}
```

**上下文相关翻译**：
- 使用 i18next 的 context 功能：`t('confirm_dialog')` vs `t('confirm_action')`
- 区分相同单词在不同上下文的含义
```json
{
  "confirm_dialog": "Confirm",
  "confirm_action": "OK"
}
```

**性别和变位**（法语、西班牙语等需要）：
- 使用 i18next 的 context 功能区分性别
- 示例：`user_male`、`user_female`

**替代方案**：
- **不处理复数和上下文**：翻译质量低，用户体验差
- **使用多个独立键值**：导致键值爆炸，难以维护

## Risks / Trade-offs

### 风险 1：硬编码文本遗漏导致部分界面未国际化

**风险**：自动化工具无法识别所有硬编码文本，导致部分界面仍显示原语言。

**缓解措施**：
- 使用多种搜索模式（中文字符、常见英文词汇、HTML 属性）
- 人工审查关键页面和组件（如聊天、设置、模型管理）
- 在测试环境进行多语言切换测试，验证覆盖范围
- 在用户反馈中持续收集和修复遗漏的文本

### 风险 2：翻译质量不一致影响用户体验

**风险**：不同语言的翻译质量参差不齐，导致用户理解困难或专业术语不统一。

**缓解措施**：
- 建立翻译规范文档，统一术语和表达方式
- 使用翻译完整性检查工具确保所有语言包含相同键值
- 优先保证英文和简体中文的翻译质量
- 引入用户反馈机制，收集翻译问题并持续改进

### 风险 3：动态文本插值导致翻译文件复杂化

**风险**：包含动态变量的文本（如"已选择 {{count}} 个项目"）在不同语言中语序不同，导致翻译困难。

**缓解措施**：
- 使用 i18next 的插值功能（`{{variable}}`）
- 保持插值变量命名一致（如 `{{count}}`、`{{name}}`）
- 在翻译文档中注明插值变量的含义和格式
- 测试动态文本的边界情况（如复数、零值）

### 风险 4：性能影响（首次切换语言时的加载延迟）

**风险**：按需加载策略可能导致用户首次切换语言时出现短暂的加载延迟。

**当前基准**（需要在准备阶段测量）：
- 英文资源包大小：~5 KB（gzipped）
- 法语资源包大小：~10 KB（gzipped）
- 首次加载时间：~500 ms（4G 网络）

**预期变化**：
- 新增 2000+ 键值后，英文资源包：~15 KB（+200%）
- 法语资源包：~30 KB（+200%）
- 首次加载时间：~1.5 s（4G 网络）

**缓解措施**：
- 复用现有的按需加载策略（英文静态加载，其他语言异步加载）
- 在切换语言时显示加载状态提示（如 Spinner）
- 使用缓存机制（已加载的语言资源缓存到内存）
- 网络失败时降级到英文并显示 Toast 警告（现有逻辑）
- **如果超出阈值**：考虑按命名空间拆分加载、使用 CDN 加速
- **性能目标**：首次加载时间 < 2 s（4G 网络）

## 翻译资源管理流程

### 开发者工作流
1. 添加新的硬编码文本时，立即在翻译文件中添加键值
2. 运行 `npm run lint:i18n` 检查完整性
3. 提交 PR 时，CI 自动检查翻译完整性
4. 确保所有 3 种语言（zh、en、fr）都包含相同键值

### 翻译工作流
1. 开发者在翻译文件中添加英文键值（作为基准）
2. UX 团队翻译中文和法文（可使用翻译管理工具辅助）
3. 使用翻译完整性检查工具确保无遗漏
4. 每次发布前，UX 团队审查翻译质量

### 质量保证
1. 建立翻译术语库，确保专业术语一致性
2. 收集用户反馈，持续改进翻译
3. 定期审查翻译文件，删除未使用的键值
4. 为关键路径添加多语言测试

### CI/CD 集成
1. 在 CI 流水线中添加 `npm run lint:i18n` 检查
2. PR 合并前自动检查所有语言的翻译完整性
3. 检查失败时阻止合并
4. 生成翻译覆盖率报告

## 测试策略

### 测试代码的处理
- **测试代码不国际化**（保持硬编码）
- **修改测试断言方式**：
  - 方案 1：使用 `data-testid` 属性断言（推荐）
  - 方案 2：在测试中 mock i18next 的 `t` 函数
- **示例**：
  ```typescript
  // 修改前
  expect(screen.getByText('新建聊天')).toBeInTheDocument();

  // 修改后（方案 1）
  expect(screen.getByTestId('new-chat-button')).toBeInTheDocument();

  // 修改后（方案 2）
  const { t } = useTranslationMock();
  t.mockReturnValue('New Chat');
  render(<Component />);
  expect(screen.getByText('New Chat')).toBeInTheDocument();
  ```

### 多语言测试
- 为关键路径添加多语言测试
- 在测试中切换语言，验证文本正确显示
- 示例：
  ```typescript
  it('should display translated text', () => {
    render(<Component />);
    expect(screen.getByText('新建聊天')).toBeInTheDocument();

    changeLanguage('en');
    expect(screen.getByText('New Chat')).toBeInTheDocument();
  });
  ```

### 翻译完整性测试
- CI 中运行 `npm run lint:i18n` 确保无遗漏
- 为 i18n 相关函数添加单元测试
- 验证所有 3 种语言的翻译文件存在且结构一致

## Migration Plan

### 阶段 1：准备阶段（1-2 天）

1. **运行硬编码文本识别脚本**
   - **第一阶段**：使用 grep/ripgrep 粗略搜索 `src/` 目录（排除 `src/__test__/`）
   - **第二阶段**：使用 AST 静态分析精确识别（减少误报到 5% 以下）
   - 人工审查识别结果，过滤误报（排除代码注释、技术标识符）
   - 生成硬编码文本清单（按文件和命名空间分类）

2. **性能基准测试**
   - 测量当前翻译资源文件大小
   - 测试首次加载时间（不同网络条件：4G、3G、WiFi）
   - 记录基准数据，便于后续对比

2. **扩展翻译资源文件**
   - 根据识别结果在 7 个命名空间中新增翻译键值
   - 为所有支持的语言（zh、en、fr）提供完整翻译
   - 运行 `npm run lint:i18n` 验证翻译完整性

- 处理动态内容（如复数、上下文相关翻译）
- 重点测试核心功能

**阶段 2.3：模型管理页面（100% 国际化）**

4. **配置 CI 检查**

### 回滚策略

**如果出现严重问题**：
- 使用 Git 回滚到变更前的版本
- 保留已扩展的翻译资源文件，便于后续修复后继续
- 在修复问题后重新实施变更

**部分回滚**：
- 如果某个命名空间的替换导致问题，可以单独回滚该命名空间的组件
