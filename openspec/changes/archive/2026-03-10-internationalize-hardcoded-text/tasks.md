# UI 文本国际化实施任务清单

## 1. 准备阶段

- [x] 1.1 运行硬编码文本识别脚本，搜索 `src/` 目录（排除 `src/__test__/`）
  - **第一阶段**：使用 grep/ripgrep 粗略搜索
    - 搜索中文字符模式：`[\u4e00-\u9fa5]+`
    - 搜索常见英文 UI 词汇：`(Save|Delete|Add|Edit|Cancel|Loading|Error|Success|Warning|Info|Send|Settings|About|Help|Create|Search|Filter|Sort)`
    - 搜索 HTML 属性中的文本：`title`、`aria-label`、`placeholder`、`alt`
  - **第二阶段**：使用 AST 静态分析精确识别
    - 使用 @babel/parser 解析 TypeScript/JSX 代码
    - 准确识别 JSX 文本节点和 HTML 属性值
    - 自动排除代码注释和技术标识符
    - 识别模板字面量中的动态内容
  - 人工审查识别结果，过滤误报
    - 排除代码注释中的文本
    - 排除技术标识符和常量（如 `ERROR_CODE = 'ERROR_FAILED'`）
    - 排除构建时的字符串拼接
  - 生成硬编码文本清单（按文件和命名空间分类）

- [x] 1.2 分析识别结果，确定需要新增的翻译键值
  - 根据 7 个命名空间（common、navigation、chat、model、setting、table、provider）分类文本
  - 确定每个键值的命名（遵循 `<namespace>.<feature>.<element>` 约定，最大 3 层）
  - 应用键值冲突解决决策树（跨组件复用 → common.*，导航 → navigation.*，特定功能 → 对应命名空间）
  - 识别动态文本插值需求（如 `{{count}}`、`{{name}}`）
  - 识别复数形式需求（如"1 个项目" vs "2 个项目"）
  - 识别上下文相关翻译需求（如"confirm_dialog" vs "confirm_action"）
  - 识别动态实体名称（如模型名称、供应商名称）

- [x] 1.3 扩展翻译资源文件（所有 3 种语言：zh、en、fr）
  - **优先级 1**：扩展 `src/locales/` 下 `common.json` 和 `navigation.json`
  - **优先级 2**：扩展 `chat.json` 和 `model.json`
  - **优先级 3**：扩展 `setting.json`、`table.json` 和 `provider.json`
  - 确保所有语言包含相同的键值结构
  - 为复数形式添加 `_plural` 后缀或语言特定的复数规则
  - 为上下文相关翻译添加 context 后缀（如 `_dialog`、`_action`）
  - 为动态实体名称添加预定义列表（如模型名称列表）
  - **优先级 1**：扩展 `src/locales/` 下 `common.json` 和 `navigation.json`
  - **优先级 2**：扩展 `chat.json` 和 `model.json`
  - **优先级 3**：扩展 `setting.json`、`table.json` 和 `provider.json`
  - 确保所有语言包含相同的键值结构

- [x] 1.4 运行翻译完整性检查
  - 执行 `npm run lint:i18n` 验证所有语言的翻译完整性
  - 修复缺失或格式错误的翻译键值
  - 确保无遗漏的翻译

- [x] 1.5 性能基准测试
  - 测量当前翻译资源文件大小（使用 `ls -lh` 或 `du -h`）
  - 测试首次加载时间（不同网络条件：4G、3G、WiFi，使用 Chrome DevTools Network Throttling）
  - 记录基准数据（文件大小、加载时间、内存占用）
  - 建立性能目标（首次加载时间 < 2 s，4G 网络）
