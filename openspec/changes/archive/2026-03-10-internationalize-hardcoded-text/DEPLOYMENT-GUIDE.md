# 国际化功能部署指南

> 版本: 1.0
> 创建日期: 2026-03-10
> 变更: internationalize-hardcoded-text

## 部署概述

本次变更为应用添加了完整的多语言支持（简体中文、英文、法语），包括：
- 扩展了7个命名空间的翻译资源（common、navigation、chat、model、setting、table、provider）
- 国际化了导航和通用组件
- 国际化了聊天、模型管理、设置等核心功能页面
- 配置了CI检查以确保翻译完整性

## 部署步骤

### 1. 创建Pull Request

```bash
# 1. 确保在正确的分支上
git status

# 2. 拉取最新代码
git pull origin main

# 3. 创建新分支（如果需要）
git checkout -b internationalize-hardcoded-text

# 4. 提交更改
git add .
git commit -m "feat: 实现UI文本国际化

- 扩展7个命名空间的翻译资源（zh/en/fr）
- 国际化导航和通用组件
- 国际化聊天、模型管理、设置页面
- 配置CI检查确保翻译完整性
- 所有测试通过（1460个测试）
"

# 5. 推送到远程
git push origin internationalize-hardcoded-text
```

### 2. 代码审查重点

审查时请特别关注：

1. **翻译文件结构**
   - 检查所有3种语言的翻译文件结构是否一致
   - 验证没有遗漏的翻译键值
   - 确认键值命名符合规范

2. **组件国际化实现**
   - 检查是否正确使用 `useTranslation()` hook
   - 验证 `t()` 函数调用格式正确
   - 确认没有遗漏的硬编码文本

3. **测试覆盖**
   - 确认i18n mock已添加到相关测试文件
   - 验证所有测试通过
   - 检查是否有新的测试失败

4. **性能影响**
   - 确认翻译文件大小增长在可接受范围
   - 验证语言切换性能正常
   - 检查内存占用是否合理

### 3. CI/CD检查验证

在PR页面确认以下检查通过：

- **ESLint**: 0个错误，0个警告

## 已知限制

1. **provider命名空间为空**
   - provider.json当前为空（0个键值）
   - provider相关翻译已整合到setting.json的modelProvider部分
   - 这是设计决策，符合避免重复的原则

2. **动态实体名称不翻译**
   - 用户自定义的模型名称保持原样
   - 供应商名称使用动态显示（provider.providerName）
   - 这是符合设计文档的

3. **控制台日志未国际化**
   - 开发者调试信息保持英文或中文硬编码
   - 符合设计文档第5章决策
