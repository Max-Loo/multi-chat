# 将硬编码文本替换为国际化配置

## Why

当前项目中存在大量硬编码的中英文文本，导致应用无法支持多语言切换，降低了用户体验和国际化能力。随着应用用户群扩大，需要支持更多语言以满足不同地区用户的需求。

## What Changes

- **识别并提取所有用户可见的硬编码文本**：从组件、页面、UI 元素中提取硬编码的中英文文本
- **扩展国际化资源文件**：在 `src/locales/` 下的相关命名空间中新增翻译键值对
- **替换硬编码文本为 i18n 调用**：将组件中的硬编码字符串替换为 `useTranslation()` 或 `t()` 函数调用
- **确保所有命名空间的翻译完整性**：为所有支持的语言（zh、en、fr）提供完整翻译
- **验证翻译完整性**：利用现有的翻译完整性检查工具确保没有遗漏

**排除范围**：
- 测试代码中的硬编码文本（`src/__test__/` 目录）
- 代码注释中的文本
- 打印到控制台的日志和警告信息
- 开发者调试信息

## Capabilities

### New Capabilities
- `i18n-ui-text`: 提供完整的 UI 文本国际化支持，包括按钮、标签、占位符、错误提示等用户可见文本的国际化能力

### Modified Capabilities
无（项目已有 i18next 基础设施和部分组件使用国际化，但仍有大量硬编码文本需要处理）

## Impact

**受影响的代码**：
- 前端组件：`src/components/`、`src/pages/` 所有包含硬编码文本的组件
- UI 组件库：`src/components/ui/` 中的基础组件
- 国际化资源：`src/locales/zh/`、`src/locales/en/`、`src/locales/fr/` 各个命名空间文件

**依赖系统**：
- 现有的 i18next + react-i18next 基础设施
- 翻译完整性检查工具：`scripts/check-i18n.js`
- 英文"第一公民"按需加载策略

**预期收益**：
- 提升用户体验：用户可以在设置中切换语言
- 便于国际化：为未来支持更多语言奠定基础
- 代码可维护性：集中管理所有用户可见文本
- 翻译质量保证：通过自动化工具确保翻译完整性
