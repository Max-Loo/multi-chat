## Context

当前 ChatButton 组件在重命名模式下，用户可以点击确认按钮提交任意内容（包括空字符串）。这会导致聊天名称为空，影响用户体验。

现有实现：
- 重命名模式由 `isRenaming` 状态控制
- 输入值存储在 `newName` 状态中
- 确认按钮直接调用 `onConfirmRename` 函数

约束条件：
- 不修改后端 API
- 不改变 Redux store 的 action 结构
- 保持组件的 memo 优化

## Goals / Non-Goals

**Goals:**
- 在输入为空时禁用确认按钮，提供视觉反馈
- 保持现有的确认逻辑（避免无意义的编辑）
- 不影响其他功能的正常使用

**Non-Goals:**
- 不修改聊天名称的验证逻辑（后端保持不变）
- 不添加额外的验证规则（如长度限制、特殊字符等）
- 不改变取消按钮的行为

## Decisions

**决策 1：使用 Button 组件的 disabled 属性**

- **选择**：通过 `disabled={!newName.trim()}` 控制确认按钮状态
- **理由**：
  - shadcn/ui 的 Button 组件原生支持 disabled 状态
  - 自动应用禁用样式（视觉反馈清晰）
  - 防止点击事件触发
- **替代方案**：在 onClick 中添加验证并显示错误提示
  - 被拒绝原因：需要额外的错误状态管理，用户体验较差

**决策 2：使用 trim() 验证空白字符**

- **选择**：验证 `newName.trim()` 是否为空
- **理由**：
  - 只包含空格的输入也视为无效
  - 与现有的"避免无意义编辑"逻辑一致（第 97 行比较 `newName === chat.name`）
- **替代方案**：只验证 `newName === ''`
  - 被拒绝原因：无法防止用户输入空格的情况

**决策 3：保持现有的确认逻辑**

- **选择**：保留第 97-100 行的"避免无意义编辑"检查
- **理由**：
  - 即使输入非空，如果名称未改变也不应该提交
  - 双重验证确保数据一致性
- **影响**：用户在未修改名称时点击确认会直接取消（现有行为）

## Risks / Trade-offs

**风险 1：禁用状态可能不够明显**
- **描述**：用户可能不清楚为什么按钮被禁用
- **缓解措施**：disabled 状态会自动应用灰色样式和不可交互效果，符合 UI 规范

**风险 2：与现有行为冲突**
- **描述**：当前如果用户清空输入框，确认按钮仍可点击
- **缓解措施**：这是改进后的预期行为，不破坏现有功能

**权衡：性能开销**
- **描述**：每次输入都会重新计算 trim() 和 disabled 状态
- **影响**：negligible（单次字符串操作，性能影响可忽略）

## Migration Plan

无迁移计划（纯前端UI改进，不需要数据迁移或部署步骤）。
