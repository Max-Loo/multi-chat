## Context

项目采用 Tauri + React + TypeScript 技术栈，前端测试基于 Vitest 框架，包含 159 个单元测试文件和 10 个集成测试文件。测试分为两套：单元测试（10s 超时）和集成测试（30s 超时，串行执行），通过 `pnpm test:all` 统一运行。

当前测试覆盖率的数字指标已经建立（各模块阈值从 65% 到 85% 不等），但尚未验证测试的实际缺陷捕获能力。变异测试是一种成熟的测试质量评估方法，通过向源码注入预设的微小错误（变异），验证测试套件能否检测到这些错误。

## Goals / Non-Goals

**Goals:**
- 量化测试套件对隐蔽逻辑错误的实际检测率（杀死率）
- 识别测试盲区——即哪些类型的 bug 可能逃过现有测试
- 针对盲区提出具体的测试改进建议
- 覆盖工具函数、加密、Redux、Hooks、服务层、存储层、配置、国际化 8 个模块

**Non-Goals:**
- 不引入任何持久性的代码变更，所有修改在流程结束后还原
- 不安装任何新的 npm 依赖或变异测试框架
- 不追求 100% 变异杀死率（本次是诊断性审计，不是全面变异测试）
- 不修改测试代码本身

## Decisions

### D1: 手动变异而非自动化工具

**选择**：手动选择 12 个变异点并逐一执行，而非使用 Stryker 等自动化变异测试框架。

**理由**：
- 自动化工具生成大量随机变异（数百到数千），运行时间过长且噪音大
- 手动选择可以精准针对「隐蔽但真实」的逻辑错误类型
- 12 个变异足以对测试质量做出有意义的评估
- 无需安装额外依赖，零基础设施成本

**备选方案**：Stryker Mutator（排除原因：配置复杂、运行时间长、对 Tauri 项目支持有限）

### D2: 变异选择策略

**选择**：按预期捕获率分层选取 12 个变异——4 高信心 + 3 中信心 + 5 低信心。

**理由**：
- 如果全部选取高信心变异，只能证明测试有效，无法暴露盲区
- 低信心变异（预期测试无法捕获）才是盲区分析的核心价值
- 混合选取能同时验证测试的有效性和发现不足

**变异类型分布**：

| 类型 | 数量 | 示例 |
|------|------|------|
| 边界条件变异 | 3 | `<=` → `<`，`Math.floor` → `Math.ceil`，`!data` → `data===null` |
| 条件逻辑变异 | 2 | 推理开关反转，trim 检查移除 |
| 缺失守卫变异 | 3 | append 失败守卫移除，loading 守卫移除，save() 调用移除 |
| React 模式变异 | 1 | useEffect 依赖项移除 |
| 枚举值变异 | 1 | `fatal` → `warning` |
| 比较/匹配变异 | 2 | hex substring 长度错误，key==translated 检查移除 |

### D3: 执行流程

**选择**：逐个变异串行执行——每次只修改一个文件，运行 `test:all`，记录结果，还原后再进行下一个。

**理由**：
- 串行执行可以精确定位每个变异对应的测试结果
- 避免多个变异互相干扰导致无法归因
- `git checkout -- <file>` 可确保完全还原

**备选方案**：并行变异（排除原因：结果归因困难，冲突风险高）

### D4: 报告格式

**选择**：Markdown 格式报告，包含每个变异的详细记录和总结分析。

**报告结构**：
1. 概览统计（总变异、杀死、存活、杀死率）
2. 每个变异的详细记录（变异内容、影响范围、测试结果、分析）
3. 盲区总结与改进建议

## Risks / Trade-offs

- **[变异样本偏差]** 12 个手动选择的变异不能代表所有可能的 bug 类型 → 缓解：确保覆盖 6 种不同的变异类型和 8 个不同的模块
- **[测试运行时间]** 每次运行 `test:all` 约需数分钟，12 次总计约 30-60 分钟 → 缓解：机器性能较好，可接受
- **[还原遗漏风险]** 如果中途意外中断，可能忘记还原某个文件 → 缓解：使用 `git checkout` 而非手动还原，可通过 `git status` 验证
- **[假阴性误判]** 某些变异可能恰好被不相关的测试捕获 → 缓解：记录具体失败的测试名称，人工分析相关性
