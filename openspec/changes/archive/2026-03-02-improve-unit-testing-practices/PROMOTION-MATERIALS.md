### 测试质量改进

**Before（测试实现细节）**：
```typescript
// ❌ 测试内部函数调用
it('应该在组件卸载时清理定时器', () => {
  const clearTimeoutSpy = vi.spyOn(global, 'clearTimeout');
  const { unmount } = renderHook(() => useDebounce('value', 500));
  unmount();
  expect(clearTimeoutSpy).toHaveBeenCalled();
});
```

**After（测试用户行为）**：
```typescript
// ✅ 测试用户可见行为
it('应该延迟更新值 当输入值变化', () => {
  vi.useFakeTimers();
  const { result, rerender } = renderHook(
    ({ value, delay }) => useDebounce(value, delay),
    { initialProps: { value: 'initial', delay: 500 } }
  );
  rerender({ value: 'updated', delay: 500 });
  expect(result.current).toBe('initial'); // 未延迟，值未更新
  act(() => vi.advanceTimersByTime(500));
  expect(result.current).toBe('updated'); // 延迟后，值已更新
  vi.useRealTimers();
});
```

**改进优势**：
- ✅ 测试关注"做什么"，而非"怎么做"
- ✅ 重构友好（改变内部实现不影响测试）
- ✅ 测试更有价值（验证实际用户行为）

## 二、新测试实践核心原则

### 1. 测试用户可见行为，而非内部实现

**判断标准**：
> **如果重命名函数、移动文件、改变内部实现，测试是否仍然通过？**

**组件测试**：
- ✅ Mock API 请求（系统边界）
- ❌ 不 Mock 子组件
- ❌ 不测试内部方法调用

**Hooks 测试**：
- ✅ 测试 Hook 返回值的行为
- ❌ 不测试内部函数调用（如 `clearTimeout` 调用次数）
- ❌ 不测试内部状态管理细节

**Redux 测试**：
- ✅ 验证用户可见的最终状态
- ❌ 不测试中间状态转换（如 `pending` → `fulfilled`）
- ✅ 更多依赖集成测试验证 Redux 行为

### 2. 仅 Mock 系统边界，不 Mock 内部实现

**系统边界（应该 Mock）**：
1. 网络 API 调用（fetch、axios、XMLHttpRequest）
2. 文件系统 I/O（fs、IndexedDB、localStorage）
3. 第三方服务（Stripe、OpenAI API）
4. 时间相关（Date、setTimeout）- 仅在必要时
5. 随机数生成（Math.random）
6. 浏览器 API（Geolocation、MediaDevices）

**内部实现（不应该 Mock）**：
1. React 组件（子组件、Hooks）
2. Redux store/selectors/actions
3. 工具函数和辅助函数
4. 内部模块和依赖

### 3. 统一测试命名规范

**格式**："应该 [预期行为] 当 [条件]"

**示例**：
- 组件测试："应该渲染错误消息 当 API 请求失败"
- Hooks 测试："应该延迟更新值 当输入值变化"
- 集成测试："应该完成完整聊天流程 当用户发送消息"

### 常见问题快速检查

| 问题 | 检查方法 | 正确做法 |
|------|----------|----------|
| **测试内部函数调用** | 查找 `vi.spyOn()` 针对内部函数 | 测试用户可见行为 |
| **Mock 子组件** | 查找 `vi.mock()` 针对 `@/components` | 渲染完整组件树 |
| **测试 Redux 中间状态** | 查找 `.pending()`、`.rejected` 测试 | 测试最终状态或使用集成测试 |
| **Mock 没有注释** | 查找无注释的 `vi.mock()` | 添加 `// Mock <module> because <reason>` |
| **测试命名不清晰** | 查找不规范的测试名称 | 使用"应该 [行为] 当 [条件]"格式 |

## 六、常见问题解答

### Q1: 为什么不 Mock 子组件？

**A**: Mock 子组件会隐藏组件间的交互问题：
- ❌ 测试不验证真实渲染结果
- ❌ 组件交互错误无法被发现
- ❌ 重构时容易失败

**更好的做法**：
```typescript
// ✅ 渲染完整组件树
render(<ChatPage />);
expect(screen.getByRole('button', { name: '发送' })).toBeInTheDocument();
```

### Q2: 所有集成测试都替代单元测试吗？

**A**: 不是。保留关键的单元测试：
- **性能关键路径**（如加密算法）
- **安全相关逻辑**（如输入验证）
- **复杂计算**（如数据处理）

**集成测试用于**：
- 验证模块间交互
- 验证完整用户流程
- 减少重复的单元测试

### Q3: 如何平衡测试速度和质量？

**A**:
- **优化慢速测试**：使用 fake timers 替代真实等待
- **并行执行**：Vitest 默认并行执行独立测试
- **监控性能**：识别和优化超过阈值的测试
- **保持阈值**：单次测试套件 < 2 分钟

### Q4: 现有测试如何迁移？

**A**: 分阶段进行：
1. **优先级排序**：先重构最脆弱的测试
2. **逐步迁移**：每次 PR 重构 1-2 个测试文件
3. **保持覆盖**：重构过程中保持测试覆盖率
4. **Code Review**：强制执行新标准

### Q5: 如何判断 Mock 是否必要？

**A**: 使用决策树：

```
是否是系统边界？
├─ 是（API、文件系统、第三方服务）
│  └─ ✅ 可以 Mock，添加注释说明理由
└─ 否（内部组件、工具函数）
   └─ ❌ 不应该 Mock
```
