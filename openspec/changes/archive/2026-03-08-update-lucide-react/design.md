# 升级 lucide-react 设计文档

## Context

lucide-react 是项目中使用的图标库，当前版本为 0.563.0。该库提供了统一的图标集和 React 组件接口，项目中的多个组件都依赖此库来显示图标。

**当前状态**：
- 版本：0.563.0
- 落后：14 个版本（最新为 0.577.0）
- 使用情况：项目中多处导入和使用 lucide-react 图标

**约束**：
- 需要确保所有现有图标在升级后仍然正常工作
- 需要验证没有破坏性变更影响现有代码
- 必须保持向后兼容性

## Goals / Non-Goals

**Goals:**
- 将 lucide-react 升级到最新版本 0.577.0
- 验证所有现有图标在升级后正常工作
- 确保没有破坏性变更影响现有功能
- 更新依赖 lockfile

**Non-Goals:**
- 重新设计图标使用方式
- 更新或修改现有图标的样式（除非破坏性变更要求）
- 添加新的图标（除非业务需求）

## Decisions

### 升级方法选择

**决策**：使用 `pnpm update lucide-react` 进行升级

**理由**：
- pnpm 的 update 命令会自动解析版本范围并安装最新的兼容版本
- 会自动更新 pnpm-lock.yaml 文件
- 比手动修改 package.json 更安全，避免版本冲突

**考虑的替代方案**：
- 手动修改 package.json 中的版本号并运行 `pnpm install`
  - 优点：更精确控制版本
  - 缺点：容易出错，可能导致依赖冲突
- 使用 `pnpm upgrade` --latest
  - 优点：确保安装绝对最新版本
  - 缺点：可能跳过重要的中间版本，增加风险

### 验证策略

**决策**：先运行类型检查和 lint，然后手动测试关键组件

**理由**：
- TypeScript 类型检查可以捕获 API 变更
- ESLint 可以发现导入和使用问题
- 手动测试可以验证视觉正确性

**考虑的替代方案**：
- 编写自动化测试覆盖所有图标使用
  - 优点：更全面的验证
  - 缺点：时间成本高，对于简单的依赖升级可能过度

### 破坏性变更处理

**决策**：如果遇到破坏性变更，记录在 tasks.md 中，并逐个修复

**理由**：
- lucide-react 的升级通常是向后兼容的
- 即使有破坏性变更，通常也很少且容易修复
- 记录问题有助于未来的参考和回滚

## Risks / Trade-offs

### 风险

**风险**：新版本可能引入未知的 bug 或行为变化
- **缓解措施**：在开发环境中充分测试，验证关键用户流程

**风险**：某些图标可能被重命名或移除
- **缓解措施**：搜索代码库中所有 lucide-react 的使用，检查是否使用了已移除的图标

**风险**：版本跳跃较大（14 个版本），可能累积多个破坏性变更
- **缓解措施**：查看 lucide-react 的 changelog，识别关键变更点

### 权衡

**开发时间 vs 风险**：
- 直接升级到最新版本可以最大化获取改进，但风险略高
- 逐个版本升级更安全，但时间成本高
- **决策**：直接升级到最新版本，因为 lucide-react 通常保持良好的向后兼容性

**测试覆盖 vs 效率**：
- 全面测试所有图标使用最安全，但耗时
- 仅测试关键路径效率高，但可能遗漏边缘情况
- **决策**：结合自动化检查（类型检查、lint）和手动验证关键组件

## Migration Plan

### 部署步骤

1. **准备工作**
   - 确保当前分支干净，所有更改已提交
   - 创建新分支用于此次升级：`git checkout -b upgrade/lucide-react-0.577.0`

2. **执行升级**
   - 运行 `pnpm update lucide-react`
   - 检查 package.json 和 pnpm-lock.yaml 的变更

3. **验证**
   - 运行 `pnpm tsc` 进行类型检查
   - 运行 `pnpm lint` 检查代码规范
   - 启动开发服务器 `pnpm tauri dev`
   - 手动测试使用图标的关键组件（如聊天界面、设置页面等）

4. **修复问题（如有）**
   - 如果遇到类型错误或 lint 错误，逐个修复
   - 如果图标被移除或重命名，更新导入和使用

5. **提交更改**
   - 验证通过后，提交更改
   - 创建 PR 进行代码审查

### 回滚策略

如果升级导致严重问题：
1. 恢复 package.json 和 pnpm-lock.yaml 到升级前的状态
2. 运行 `pnpm install` 恢复旧版本依赖
3. 如果已提交，使用 `git revert` 回滚提交
