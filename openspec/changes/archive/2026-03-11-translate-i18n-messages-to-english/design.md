# 翻译 i18n.ts 中的硬编码消息为英文

## Context

**当前状态**：`src/lib/i18n.ts` 作为国际化系统的核心模块，内部存在多处硬编码的中文字符串（Toast 消息和 console 日志），这与国际化系统的职责定位不符。

**问题**：在 i18n 初始化阶段无法使用 `t()` 函数（鸡生蛋问题 - i18n 未初始化完成时无法调用翻译函数），因此需要一种不依赖 i18n 系统的解决方案。

**约束条件**：
- 不能在 i18n 初始化完成前使用 `t()` 函数
- 不能改变外部 API 和功能行为
- 不能增加新的依赖或复杂度
- 需要保持代码的一致性和可维护性

## Goals / Non-Goals

**Goals:**
- 移除 `src/lib/i18n.ts` 中所有硬编码的中文字符串
- 统一使用英文作为初始化阶段的提示语言
- 避免国际化初始化依赖问题
- 保持代码简洁，不引入额外复杂度

**Non-Goals:**
- 不改变外部 API 接口
- 不影响功能行为
- 不引入新的翻译系统或工具
- 不改变其他模块的字符串处理方式

## Decisions

**决策 1：将硬编码中文字符串翻译成英文**

- **选择理由**：
  1. 英文是项目的 fallback 语言，合理使用英文不会造成逻辑冲突
  2. 实现最简单直接，无需引入新的机制或依赖
  3. 统一使用英文，降低代码维护复杂度
  4. 避免了鸡生蛋问题（不需要在 i18n 初始化前使用翻译函数）

- **替代方案考虑**：
  - *方案 A：创建预初始化翻译辅助函数* - 虽然更灵活，但会增加维护成本和复杂度，对于仅 7 处字符串的场景属于过度设计
  - *方案 B：延迟显示 Toast* - 会影响用户体验，初始化阶段的提示信息应当及时反馈
  - *方案 C：保持中文* - 与国际化系统的职责定位不符，也会导致代码不一致

**决策 2：直接修改字符串，不引入新的抽象层**

- **选择理由**：
  1. 变更范围很小（约 7 处字符串），不值得引入新的抽象
  2. 保持代码简洁直观
  3. 易于后续维护和理解

- **实施范围**：
  - Toast 消息（语言迁移提示、系统语言切换提示、错误提示等）
  - Console 日志（警告、错误信息）

## Risks / Trade-offs

**风险 1：用户体验变化**
- **描述**：中文用户在初始化阶段看到的提示将变为英文
- **缓解措施**：
  1. Toast 消息使用清晰简洁的英文，易于理解
  2. 关键功能行为不受影响，仅提示语言变化
  3. 初始化完成后，所有用户界面仍会使用用户选择的语言

**风险 2：开发者可读性**
- **描述**：中文开发者可能需要适应英文日志
- **缓解措施**：
  1. 英文是技术社区的标准语言
  2. 使用标准化的英文错误消息格式
  3. 代码注释保持中文（符合项目约定）

**权衡：简单性 vs 多语言支持**
- **选择**：优先简单性
- **理由**：初始化阶段的多语言支持价值有限，而实现复杂度显著增加。英文作为 fallback 语言已足够使用。

## Migration Plan

**实施步骤**：
1. 定位 `src/lib/i18n.ts` 中所有硬编码的中文字符串
2. 将每个中文字符串翻译成对应的英文
3. 保持字符串插值和格式一致
4. 测试验证：
   - 验证 Toast 消息正确显示
   - 验证 console 日志正确输出
   - 验证语言迁移、切换逻辑正常工作
   - 验证错误场景的提示信息

**回滚策略**：
- 如有问题，可通过 git revert 快速回滚
- 变更范围仅限单个文件，回滚风险低
