## Context

当前 `ChatPanelContent` 组件在非 splitter 模式下渲染网格布局，每个单元格都固定添加 `border-b` 和 `border-r`。这导致：
- 最右侧单元格的右边框溢出到容器外边缘
- 最底行单元格的下边框溢出到容器底部

期望效果：只有相邻单元格之间才有分隔线，形成清晰的网格边界。

## Goals / Non-Goals

**Goals:**
- 实现条件性边框渲染逻辑
- 保持代码简洁易维护
- 正确处理不规则网格（最后一行不满的情况）

**Non-Goals:**
- 不修改 ResizablePanel 模式（splitter=true）的行为
- 不改变网格布局算法
- 不修改边框样式（颜色、宽度等）

## Decisions

### 决策 1：使用条件判断方案（方案 A）

**选择**：在渲染时根据单元格位置动态决定边框类名

**理由**：
- 改动最小，只需修改一个文件
- 逻辑清晰，易于理解和维护
- 不引入新的 CSS 技巧或依赖

**备选方案**：
- **方案 B（负边距叠加）**：使用 `margin: -1px` 让边框在间隙中重叠，外边缘被容器裁剪。优点是不需要条件判断，缺点是引入额外复杂度，可能影响布局计算。
- **方案 C（CSS Grid + gap）**：重构为 Grid 布局，用 `gap` + 伪元素画线。优点是语义更现代，缺点是需要重构整个布局逻辑。

### 决策 2：使用 `cn` 工具函数构建类名

**选择**：使用项目已有的 `cn` 工具函数（来自 `@/lib/utils`）

**理由**：
- 代码更简洁，避免字符串拼接
- 与项目现有代码风格一致
- 自动处理条件值为 false 的情况

**实现方式**：
```tsx
className={cn(
  "relative flex-1 min-w-0 border-gray-300",
  cellIdx < row.length - 1 && "border-r",
  idx < board.length - 1 && "border-b"
)}
```

## Risks / Trade-offs

**风险 1：不规则网格的边界判断**
- **风险**：最后一行单元格数量少于 `columnCount` 时，边框逻辑可能不正确
- **缓解**：当前逻辑使用 `row.length` 而非 `columnCount`，已正确处理此情况

**风险 2：边框闪烁或重绘**
- **风险**：当 `columnCount` 变化时，边框可能闪烁
- **缓解**：这是 React 正常的重新渲染行为，影响可忽略

**权衡**：
- 选择条件判断而非纯 CSS 方案，增加了少量逻辑，但提升了代码可读性
- 保持边框样式（`border-gray-300`）不变，仅修改显示逻辑
