## Context

**当前状态**: 项目使用 `pnpm` 管理依赖，当前 `package.json` 中存在 **38 个过时的依赖包**。

**问题背景**: 
- 长时间未更新依赖导致潜在的安全漏洞和 bug 修复无法获得
- 依赖生态系统需要保持健康和同步
- 技术债务会随时间累积

**约束条件**:
- 采用**分阶段更新策略**以降低风险
- 阶段 1 仅更新**补丁版本**（patch versions），即第三位版本号变更
- 不能破坏现有功能
- 必须能够快速回滚

**利益相关者**: 开发团队、项目维护者

## Goals / Non-Goals

**Goals:**
- 批量更新 38 个补丁版本依赖到最新稳定版本
- 确保所有测试通过（单元测试 + 集成测试）
- 保持代码质量（lint 和类型检查通过）
- 文档化更新过程以便后续阶段参考

**Non-Goals:**
- 不涉及应用代码的修改（除非依赖有破坏性变更）
- 不更新次版本或主版本依赖（留待阶段 2 和 3）
- 不引入新的依赖

## Decisions

### 决策 1: 使用批量更新而非逐个更新

**选择**: 使用 `pnpm update` 批量更新所有补丁版本

**理由**:
- 补丁版本通常只包含 bug 修复和安全补丁，不包含破坏性变更
- 批量更新效率更高，节省时间
- 所有依赖都是补丁版本，风险可控

**替代方案**:
- **逐个更新**: 更安全但耗时过长（38 个包）
- **选择性更新**: 可能遗漏重要的安全修复

### 决策 2: 不锁定依赖版本

**选择**: 保持现有的版本范围配置（如 `^19.2.0`）

**理由**:
- 项目已使用 `^` 范围，保持一致性
- 允许自动获得补丁版本更新
- `pnpm-lock.yaml` 已锁定确切版本

**替代方案**:
- **锁定确切版本**: 更严格但失去自动补丁更新的便利

### 决策 3: 测试优先的更新策略

**选择**: 更新后立即运行完整测试套件

**理由**:
- 尽早发现潜在问题
- 确保更新不破坏现有功能
- 建立测试基线用于后续阶段

**实施顺序**:
1. 更新依赖
2. 运行类型检查 (`pnpm tsc`)
3. 运行 lint (`pnpm lint`)
4. 运行测试 (`pnpm test:all`)
5. 手动验证核心功能

## Risks / Trade-offs

### 风险 1: 隐藏的破坏性变更
**风险**: 即使是补丁版本，极少数情况下可能包含行为变更

**缓解措施**:
- 更新前仔细审查 `pnpm outdated` 输出
- 运行完整测试套件验证
- 如有异常，可通过 Git 快速回滚

### 风险 2: 传递依赖冲突
**风险**: 更新的依赖可能引入新的传递依赖，导致冲突

**缓解措施**:
- `pnpm` 的依赖解析算法会自动处理冲突
- 检查 `pnpm-lock.yaml` 的变更
- 如有问题，查看具体的依赖解析结果

### 风险 3: 测试覆盖不足
**风险**: 测试可能无法覆盖所有受影响的功能

**缓解措施**:
- 手动验证核心功能（聊天、模型管理、设置等）
- 检查控制台是否有警告或错误
- 在阶段 1 中积累经验，改进阶段 2 和 3 的测试策略

## Migration Plan

### 实施步骤

1. **更新依赖**
   ```bash
   # 批量更新所有补丁版本
   pnpm update
   ```

2. **验证更新**
   ```bash
   # 检查变更
   git diff package.json
   git diff pnpm-lock.yaml
   
   # 类型检查
   pnpm tsc
   
   # Lint
   pnpm lint
   
   # 运行测试
   pnpm test:all
   ```

3. **手动测试**
    - 启动应用: `pnpm web:dev`
    - 测试核心功能:
      - 发送和接收消息
      - 切换模型
      - 管理聊天列表
      - 修改设置
      - 切换语言
   ```

### 回滚策略

如遇问题，可通过以下方式快速回滚：

**方案 1: 选择性回滚单个包**
```bash
# 如果只有特定包有问题
pnpm update <package-name>@<old-version>
```

**方案 2: 恢复更新前的依赖文件**
```bash
# 恢复 package.json 和 pnpm-lock.yaml
# （需要通过 Git 或其他方式恢复到更新前的版本）
git checkout <pre-update-commit> -- package.json pnpm-lock.yaml
pnpm install
```

### 验证清单

- [ ] 所有测试通过 (`pnpm test:all`)
- [ ] 类型检查通过 (`pnpm tsc`)
- [ ] Lint 通过 (`pnpm lint`)
- [ ] 应用正常启动 (`pnpm web:dev`)
- [ ] 核心功能正常（聊天、模型管理、设置）
- [ ] 控制台无警告或错误
- [ ] `package.json` 变更符合预期（仅补丁版本）
- [ ] `pnpm-lock.yaml` 已正确更新

## Open Questions

**问题 1: 是否需要为阶段 2 和 3 创建功能分支？**

**待决定**: 是在阶段 1 完成后直接继续，还是为每个阶段创建独立的功能分支？

**建议**: 为每个阶段创建独立分支，便于单独测试和回滚。

**问题 2: 如何处理更新后发现的问题？**

**待决定**: 如果在更新后发现测试失败或功能异常，是立即回滚还是尝试修复？

**建议**:
- 补丁版本更新：优先回滚，避免浪费调试时间
- 次版本和主版本更新：可考虑修复，因为有更大的价值

**问题 3: 是否需要更新文档？**

**待决定**: 依赖更新是否需要更新 README、CHANGELOG 或其他文档？

**建议**: 
- 阶段 1: 不需要（补丁版本不用户可见）
- 阶段 2 和 3: 如有用户可见的功能变更，需要更新文档

## Performance Considerations

**预期影响**:
- **构建时间**: 可能略微减少（依赖优化）
- **运行时性能**: 可能略有提升（bug 修复和优化）
- **Bundle 大小**: 基本无变化（补丁版本通常不影响代码体积）

**监控指标**:
- 应用启动时间
- 首屏渲染时间
- 内存占用
- Bundle 大小（使用 `rollup-plugin-visualizer`）

## Security Considerations

**安全更新**:
- 补丁版本通常包含安全修复
- 审查 `pnpm audit` 输出以确认安全漏洞已修复

**验证步骤**:
```bash
# 更新前检查
pnpm audit

# 更新后再次检查
pnpm audit

# 确认高严重性漏洞已修复
```
