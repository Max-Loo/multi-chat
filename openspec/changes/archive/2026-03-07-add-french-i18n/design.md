## Context

**当前状态**
- 应用使用 i18next + react-i18next 进行国际化管理
- 目前支持中文 (`zh`) 和英文 (`en`) 两种语言
- 翻译文件位于 `src/locales/{langCode}/` 目录下，按功能模块分割（common、chat、navigation 等）
- 语言配置通过 `SUPPORTED_LANGUAGE_LIST` 常量管理
- 用户语言偏好保存在 localStorage 中
- i18next 通过 `import.meta.glob` 动态加载所有 `src/locales/**/*.json` 文件

**约束条件**
- 不得破坏现有中文和英文用户的体验
- 必须保持现有的文件结构和加载机制
- 翻译质量必须达到可用标准
- 无需添加新的依赖项（继续使用现有的 i18next）

**相关方**
- 法语用户：需要完整的法语界面体验
- 开发团队：需要可维护的翻译文件和清晰的工作流程
- QA 团队：需要验证法语界面的正确性

## Goals / Non-Goals

**Goals:**
- 为应用添加完整的法语国际化支持
- 确保所有 UI 文本都有对应的高质量法语翻译
- 保持与现有国际化架构的一致性
- 提供清晰、可维护的翻译文件结构

**Non-Goals:**
- 不修改现有的国际化加载机制或架构
- 不添加新的第三方翻译服务或工具
- 不实现翻译的自动更新或热重载功能
- 不涉及 RTL（从右到左）布局支持（法语为 LTR 语言）

## Decisions

### 决策 1：翻译方法采用 AI 翻译 + 人工审核

**选择**：使用 AI 翻译工具（如 Claude、GPT）翻译所有文本，然后由人工进行审核和调整。

**理由**：
- **效率**：121 个翻译条目，纯人工翻译需要 1-2 小时，AI 翻译可在几分钟内完成
- **质量可控**：UI 文本翻译相对标准，AI 能够提供高质量的初稿
- **成本效益**：人工审核比完全人工翻译节省时间
- **一致性**：可以要求 AI 使用统一的翻译风格和术语

**替代方案**：
- **完全人工翻译**：质量最高但耗时较长
- **纯 AI 翻译无审核**：速度快但可能存在术语不一致或语法问题

**实施方式**：
1. 使用 AI 工具批量翻译英文版本的翻译文件
2. 提供标准术语表（如 "Model Provider" → "Fournisseur de Modèles"）
3. 要求 AI 保留所有插值变量（如 `{{count}}`）
4. 人工审核每个翻译文件，重点关注：
   - UI 术语的准确性
   - 变量插值的正确性
   - 法语语法和拼写
   - 文本长度（避免过长的翻译破坏布局）

### 决策 2：保持现有文件结构和命名规范

**选择**：完全复刻现有的英文翻译文件结构，创建对应的法语版本。

**理由**：
- **一致性**：与现有架构保持一致，降低维护成本
- **自动化加载**：`import.meta.glob('../locales/**/*.json')` 会自动发现新文件
- **模块化**：按功能分割的文件结构便于维护和查找

**文件结构**：
```
src/locales/fr/
├── common.json      # 通用 UI 文本
├── chat.json        # 聊天相关
├── navigation.json  # 导航菜单
├── setting.json     # 设置页面
├── model.json       # 模型管理
├── table.json       # 表格组件
└── provider.json    # 供应商信息
```

### 决策 3：处理法语语法特性（性数配合）

**选择**：在翻译中正确处理法语的性数配合，特别是涉及计数的场景。

**理由**：
- 法语有严格的语法规则，名词和形容词需要根据数量和性别进行配合
- 英语使用复数形式（如 "models"），法语需要配合（如 "modèles trouvés"）
- i18next 支持通过变量传递和上下文处理语法变化

**实施策略**：
1. **使用简化表达避免语法配合问题**：由于 i18next 的简单插值无法根据变量值自动调整性数配合，采用中性表达（如使用冒号、避免形容词配合）
2. 保留所有插值变量的位置和格式
3. 对于必须处理复数的场景，可考虑使用 i18next 的复数功能（`_plural` 后缀）

**示例（推荐方案 - 中性表达）**：
```json
// English
"modelCount": "{{count}} models",
"searchResult": "Found {{count}} models",

// French (中性表达，避免配合问题)
"modelCount": "{{count}} modèles",
"searchResult": "Résultat : {{count}} modèles",
```

**备选方案（使用 i18next 复数功能）**：
```json
// French (复数处理)
"modelCount": "{{count}} modèle",
"modelCount_plural": "{{count}} modèles",
"searchResult": "{{count}} modèle trouvé",
"searchResult_plural": "{{count}} modèles trouvés",
```

### 决策 4：语言选择器 UI 更新

**选择**：在 `LANGUAGE_OPTIONS` 数组中添加法语选项，使用国旗 emoji 和法语原生名称。

**理由**：
- 符合现有的语言选项格式（如 "🇨🇳 中文"、"🇺🇸 English"）
- 国旗 emoji 提供视觉识别，原生名称确保准确性
- 按字母顺序排列（中文、English、Français）或按语言代码顺序

**实现**：
```typescript
const LANGUAGE_OPTIONS = [
  { value: "zh", label: "🇨🇳 中文" },
  { value: "en", label: "🇺🇸 English" },
  { value: "fr", label: "🇫🇷 Français" },
] as const;
```

### 决策 5：不使用法语特有字符处理

**选择**：使用标准的 UTF-8 编码，不进行特殊的字符处理或规范化。

**理由**：
- 法语使用的扩展 Latin 字符（é, è, ê, ç, à 等）在 UTF-8 中完全支持
- JSON 和现代浏览器/编辑器原生支持这些字符
- 现有的中文和英文翻译文件也使用 UTF-8，无需特殊处理

**注意事项**：
- 确保所有翻译文件保存为 UTF-8 编码（无 BOM）
- 确保代码编辑器和 IDE 正确显示法语字符

## Risks / Trade-offs

### 风险 1：翻译质量问题
**风险描述**：AI 翻译可能在某些上下文中不够准确或自然。

**缓解措施**：
- 人工审核所有翻译内容
- 提供标准术语表给 AI 工具
- 重点审核 UI 术语（如 "Settings"、"Cancel" 等）
- 可邀请法语使用者进行用户体验测试

### 风险 2：文本长度导致布局问题
**风险描述**：法语翻译可能比英文长 20-30%，可能导致 UI 组件溢出或换行。

**缓解措施**：
- 在审核阶段检查翻译长度，尽量使用简洁的表达
- 测试法语界面，检查是否有布局问题
- 如有需要，可考虑 CSS 调整（但应尽量避免）

**示例**：
- English: "Delete" (6 字符)
- French: "Supprimer" (10 字符)

### 风险 3：插值变量语法错误
**风险描述**：AI 翻译可能遗漏或错误放置插值变量（如 `{{count}}`）。

**缓解措施**：
- 在 AI prompt 中明确要求保留所有变量
- 人工审核时逐一验证插值变量的存在和位置
- 可编写脚本自动验证键名一致性

### 风险 4：不一致的术语翻译
**风险描述**：同一术语在不同文件中可能被翻译为不同的法语词汇。

**缓解措施**：
- 建立术语表，统一关键术语的翻译：
  - "Model" → "Modèle"
  - "Provider" → "Fournisseur"
  - "Chat" → "Chat" 或 "Conversation"（需要决策）
  - "Settings" → "Paramètres"
- 人工审核时交叉检查相关文件

## Open Questions

### 问题 1：插值变量的语法配合方案
**问题**：法语有严格的性数配合规则，但 i18next 的简单插值 `{{count}}` 无法自动根据值调整语法。如何处理？

**方案 A（推荐）**：使用中性表达，避免形容词配合
- 示例：`"Résultat : {{count}} modèles"` 而非 `"{{count}} modèles trouvés"`
- 优点：实现简单，无需特殊处理
- 缺点：表达可能不如自然法语流畅

**方案 B**：使用 i18next 的复数功能（`_plural` 后缀）
- 示例：创建 `"searchResult"` 和 `"searchResult_plural"` 两个键
- 优点：语法正确，更自然
- 缺点：需要为每个涉及复数的翻译创建两个键

**待决策**：实施前需要明确选择方案 A 或方案 B。

### 问题 2："Chat" 的法语翻译
**问题**：在 UI 中，"Chat" 应该翻译为 "Chat"（保留英文，常见于法语应用）还是 "Conversation"（更传统的法语）？

**建议**：使用 "Chat"，因为：
- 在现代法语应用中很常见
- 与代码中的术语保持一致
- 更简洁

**待决策**：需要团队确认或参考竞品。

### 问题 3：是否需要添加加拿大法语 (fr-CA) 变体
**问题**：法语在欧洲 (fr-FR) 和加拿大 (fr-CA) 有一些差异，是否需要分别支持？

**建议**：初期只支持通用法语 (fr)，主要使用欧洲法语。
- 法语用户的国际化需求相对标准
- 如有特定地区需求，后续可基于 fr 创建 fr-CA 变体

**待决策**：根据用户反馈决定是否添加地区变体。

### 问题 4：翻译文件的版本控制
**问题**：如何管理翻译文件的更新？当英文版本新增键时，如何同步到法语？

**建议**：
- 在提交代码前，确保法语翻译文件与英文文件的键完全一致
- 可编写脚本自动验证键名一致性（已在任务 6.1 中添加）
- 未来可考虑使用翻译管理工具（如 i18next-locize-backend）

**待决策**：是否需要自动化工具来维护翻译一致性。
