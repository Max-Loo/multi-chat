## Context

当前测试系统中存在两类问题：

1. **测试归属错误**：`drawer-state.integration.test.tsx` 中行 104 和行 124 的两个测试仅操作 Redux store（dispatch + getState），不涉及任何 DOM 断言，却渲染了完整页面组件，属于单元测试范畴。
2. **死代码**：4 个 mock 工厂文件中有 7 个导出函数未被任何测试文件导入使用，增加代码库维护负担。

涉及文件均位于 `src/__test__/` 目录下，变更范围可控。

## Goals / Non-Goals

**Goals:**

- 将 2 个 Redux 纯逻辑测试迁移到正确的 slice 单元测试文件中
- 删除 7 个已确认无消费者的 mock 导出函数
- 保持现有测试覆盖率不降低

**Non-Goals:**

- 不重构其他测试文件
- 不修改 mock 工厂的整体架构
- 不涉及 P2/P3 优先级的问题

## Decisions

### D1: 迁移目标文件选择

**决策**：迁移到已有的 slice 单元测试文件（如 `chatPageSlices.test.ts` 或各自 slice 的测试文件），而非创建新文件。

**理由**：这两个测试验证的是跨 slice 的抽屉状态独立管理逻辑，涉及 `chatToggleDrawer`、`settingToggleDrawer` 等 action，与 `chatPageSlices.test.ts` 已有测试属于同一范畴。

**备选方案**：
- 创建独立的 `drawer-state.test.ts` → 不必要，测试数量少，不值得新建文件

### D2: 迁移后清理集成测试文件

**决策**：从集成测试文件中直接删除这 2 个测试，同时清理仅被这些测试使用的导入。

**理由**：避免测试逻辑重复，保持集成测试专注于真正的集成场景（组件渲染 + Redux 交互）。

### D3: 死代码删除策略

**决策**：直接删除未使用的导出函数，保留文件中其他仍在使用的导出。

**理由**：通过 grep 已确认无任何消费者，删除是安全的。`createCryptoMocks` 虽有自测文件，但自测测的是一个无消费者的函数，应一并删除。

## Risks / Trade-offs

- **[低风险] 迁移测试可能引入微小差异** → 迁移是复制 + 调整导入路径，逻辑不变，风险极低
- **[低风险] 删除 mock 导出后未来可能需要重新实现** → Git 历史可恢复，重新实现成本极低（每个函数 < 20 行）
