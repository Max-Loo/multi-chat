# 本地日志采集系统 - 技术设计

## Context

### 背景

multi-chat 是一个 Tauri 2.0 + React 19 桌面应用，目前没有日志系统。当用户遇到问题时，开发者无法获取足够的上下文信息来诊断问题。

### 当前状态

- 前端使用 `console.log/error/warn` 进行调试输出
- 后端 Rust 没有任何日志输出
- 没有错误追踪机制
- 没有日志持久化

### 约束

- 应用没有后端服务器，无法实现远程日志上报
- 日志存储在用户本地设备
- 需要保护用户隐私，敏感信息必须脱敏
- 不能显著影响应用性能

### 相关方

- **用户**：需要简单易用的日志导出功能
- **开发者**：需要足够的日志信息来排查问题

## Goals / Non-Goals

**Goals:**

1. 统一的前端日志接口，支持多级别日志
2. 前端日志通过 IPC 传递到 Rust 后端统一存储
3. 按天轮转日志文件，保留 30 天
4. 自动脱敏敏感信息（API Key、聊天内容、个人信息）
5. 支持 JSON 格式导出日志
6. 开发/生产模式行为区分

**Non-Goals:**

1. 远程日志上报（没有后端服务器）
2. 应用内日志查看器（只需导出功能）
3. 实时日志监控
4. 结构化日志查询（如 ELK）

## Decisions

### D1: 日志架构 - 混合模式

**决策**：前端 Logger 通过 `invoke()` 调用 Rust 命令写入日志，统一存储在文件系统。

```
┌─────────────────────────────────────────────────────────────────────┐
│                         日志架构                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   React 前端                              Rust 后端                  │
│  ┌─────────────────┐                    ┌─────────────────┐        │
│  │ Logger.ts       │                    │ log_write()     │        │
│  │ + Sanitizers    │ ── invoke() ────▶  │ log_export()    │        │
│  │                 │                    │ log_clear()     │        │
│  │ Transports:     │                    │                 │        │
│  │ - TauriTransport│                    │ tracing +       │        │
│  │ - ConsoleTrans. │                    │ tracing-appender│        │
│  └─────────────────┘                    └────────┬────────┘        │
│                                                  │                  │
│                                                  ↓                  │
│                                         ┌─────────────────┐        │
│                                         │ 日志文件存储     │        │
│                                         │ (按天轮转)       │        │
│                                         └─────────────────┘        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**备选方案**：
- **方案 A（纯前端）**：仅使用 IndexedDB 存储 - 无法捕获 Rust 层错误
- **方案 B（纯后端）**：前端不记录，仅 Rust 记录 - 前端错误需要手动传递

**选择理由**：混合模式可以统一采集前后端日志，存储位置一致，便于导出和管理。

### D2: 日志存储格式 - JSON Lines

**决策**：使用 JSON Lines 格式（每行一个 JSON 对象）。

```json
{"timestamp":"2026-03-15T10:30:45.123Z","level":"INFO","source":"frontend","message":"Chat sent","context":{"provider":"openai"}}
{"timestamp":"2026-03-15T10:30:45.456Z","level":"ERROR","source":"backend","message":"API request failed","context":{"error":"timeout"}}
```

**备选方案**：
- **纯文本格式**：难以解析和查询
- **二进制格式**：不利于调试和用户查看

**选择理由**：JSON Lines 兼顾可读性和机器解析能力，导出时可直接合并为数组。

### D3: 敏感信息脱敏 - 前端预处理

**决策**：在前端 Logger 中进行脱敏处理，再传递给 Rust 写入文件。

```
┌─────────────────────────────────────────────────────────────────────┐
│                       脱敏管道                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   原始日志对象                                                       │
│        │                                                            │
│        ↓                                                            │
│  ┌───────────────────┐                                             │
│  │ ApiKeySanitizer   │  sk-xxxxxxxxxxxx → sk-xxxx****xxxx          │
│  └───────────────────┘                                             │
│        │                                                            │
│        ↓                                                            │
│  ┌───────────────────┐                                             │
│  │ ContentSanitizer  │  长文本 → [CONTENT: 256B]                   │
│  └───────────────────┘                                             │
│        │                                                            │
│        ↓                                                            │
│  ┌───────────────────┐                                             │
│  │ PiiSanitizer      │  邮箱/手机 → [EMAIL]/[PHONE]                │
│  └───────────────────┘                                             │
│        │                                                            │
│        ↓                                                            │
│   脱敏后日志对象 → invoke('log_write')                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**脱敏规则**：

| 数据类型 | 匹配规则 | 脱敏结果 |
|---------|---------|---------|
| API Key | 字段名含 apiKey/api_key/token | 保留前4后4，中间 `****` |
| 聊天内容 | 字段名为 content/message | `[CONTENT: xxxB]` |
| 邮箱 | 正则 `/\S+@\S+\.\S+/` | `[EMAIL]` |
| 手机号 | 正则 `/\d{11}/` | `[PHONE]` |
| 文件路径 | 包含用户目录的路径 | `~/.../filename` |

**脱敏边界情况**：
- **递归遍历**：脱敏器 SHALL 递归遍历对象的所有层级，检查并脱敏匹配的字段
- **短内容处理**：聊天内容无论长短，只要字段名匹配就进行脱敏（移除"超过 50 字符"的限制）
- **数组处理**：数组中的每个元素 SHALL 同样进行脱敏检查

**备选方案**：
- **后端脱敏**：需要在前端传递敏感信息到 Rust，增加暴露风险

**选择理由**：前端脱敏确保敏感信息永不离开 JavaScript 运行环境。

### D4: 日志轮转策略 - 按天轮转

**决策**：使用 tracing-appender 实现按天轮转，保留 30 天。

**文件命名**：
- 当前日志：`multi-chat.log`
- 轮转后：`multi-chat.2026-03-15.log`

**清理策略**：每天凌晨检查并删除超过 30 天的日志文件。

**备选方案**：
- **按大小轮转**：单文件超 10MB 后轮转 - 文件管理复杂
- **不轮转**：单个文件持续增长 - 存储空间不可控

**选择理由**：按天轮转便于按时间范围导出，实现简单。

### D5: 开发/生产模式区分

**决策**：通过 Tauri 的 `dev` 特性标志区分行为。

| 行为 | 开发模式 | 生产模式 |
|-----|---------|---------|
| 控制台输出 | ✓ 启用 | ✗ 禁用 |
| 日志级别 | DEBUG+ | INFO+ |
| 脱敏 | 可选（方便调试） | 强制 |
| 文件写入 | 可选 | ✓ 启用 |

**实现方式**：
- 前端：通过 `import.meta.env.DEV` 判断
- Rust：通过 `#[cfg(debug_assertions)]` 判断

### D6: 文件结构

**决策**：按以下结构组织代码。

```
src/
├── utils/logger/                    # 前端日志模块
│   ├── index.ts                     # 主入口，导出 logger 实例
│   ├── Logger.ts                    # Logger 类
│   ├── types.ts                     # 类型定义
│   ├── transports/
│   │   ├── TauriTransport.ts        # 调用 Rust 写入
│   │   └── ConsoleTransport.ts      # 开发模式控制台输出
│   └── sanitizers/
│       ├── index.ts                 # 统一脱敏管道
│       ├── apiKey.ts                # API Key 脱敏
│       ├── content.ts               # 内容脱敏
│       └── pii.ts                   # 个人信息脱敏
│
└── pages/Setting/components/
    └── LogSettings/                 # 日志设置组件
        └── index.tsx                # 导出/清除按钮

src-tauri/src/
├── commands/
│   └── log.rs                       # 日志相关 Tauri 命令
└── logging/
    ├── mod.rs                       # 模块导出
    ├── config.rs                    # 日志配置
    └── writer.rs                    # 文件写入器
```

### D7: Tauri 命令接口定义

**log_status 命令返回格式**：

```json
{
  "totalSizeBytes": 12345678,
  "fileCount": 5,
  "oldestDate": "2026-02-13",
  "newestDate": "2026-03-15",
  "retentionDays": 30
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| totalSizeBytes | number | 日志文件总大小（字节） |
| fileCount | number | 日志文件数量 |
| oldestDate | string | 最早日志日期（YYYY-MM-DD） |
| newestDate | string | 最新日志日期（YYYY-MM-DD） |
| retentionDays | number | 保留天数配置 |

**log_write 命令参数**：

```json
{
  "level": "INFO",
  "message": "User action completed",
  "context": { "action": "send_chat" },
  "source": "frontend"
}
```

**log_export 命令参数**：

```json
{
  "format": "json"
}
```

返回：文件路径（用于触发下载）

**log_clear 命令**：

无参数，返回清除的文件数量和释放的空间大小。

## Risks / Trade-offs

### R1: IPC 调用开销

**风险**：高频日志调用可能影响性能（每次 `logger.info()` 都是一次 IPC 调用）。

**缓解**：
- 前端实现批量缓冲，每 100ms 或累积 10 条日志批量发送
- 生产环境仅记录 INFO 及以上级别

### R2: 脱敏遗漏

**风险**：脱敏规则可能遗漏某些敏感信息格式。

**缓解**：
- 提供可配置的脱敏规则
- 在代码审查中检查日志调用
- 导出前二次检查日志内容

### R3: 存储空间

**风险**：30 天日志可能占用较大存储空间（预估 30-150MB）。

**缓解**：
- 在设置页面显示当前日志大小
- 允许用户手动清除日志
- 单文件超 10MB 时压缩存储（后续优化）

### R4: 日志文件损坏

**风险**：应用崩溃可能导致日志文件不完整。

**缓解**：
- 每条日志立即 flush 到磁盘
- 使用 JSON Lines 格式，即使文件不完整也可解析

## Migration Plan

### 阶段 1: 基础设施（Phase 1）

1. 添加 Rust 依赖（tracing, tracing-subscriber, tracing-appender）
2. 创建日志配置模块
3. 实现文件写入器
4. 实现 Tauri 命令（log_write, log_export, log_clear）
5. 在 lib.rs 中初始化日志系统

### 阶段 2: 前端日志器（Phase 2）

1. 创建类型定义
2. 实现脱敏器
3. 实现 Transports
4. 实现 Logger 类
5. 集成全局错误捕获

### 阶段 3: UI 集成（Phase 3）

1. 创建 LogSettings 组件
2. 集成到设置页面
3. 实现导出功能
4. 实现清除功能

### 阶段 4: 日志埋点（Phase 4）

1. 迁移现有 console.* 到 Logger
2. 为关键流程添加日志埋点
3. 改进错误处理（添加上下文）

### 回滚策略

- 日志系统是独立模块，可随时禁用
- 禁用后应用正常运行，只是没有日志记录
- 用户可手动删除日志文件夹

## Open Questions

1. **日志级别可配置性**：是否允许用户在设置中调整日志级别？（如从 INFO 改为 DEBUG）
   - 当前决策：暂不支持，使用固定配置
   - 后续可考虑作为高级选项

2. **自动错误报告**：未来如果添加后端服务，是否支持自动上传崩溃日志？
   - 当前决策：不实现，但设计上预留扩展点
