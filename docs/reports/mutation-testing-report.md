### M06 — hex 验证条件反转

| 属性 | 详情 |
|------|------|
| **模块** | `src/utils/crypto.ts` — `hexToBytes` |
| **变异类型** | 条件反转 |
| **变异内容** | `hex.length % 2 !== 0` → `hex.length % 2 === 0` |
| **预期信心** | 高 |
| **实际结果** | 已杀死 |
| **失败测试数** | 52/79 |

**失败的测试用例**：级联失败，`hexToBytes` 是加密模块的基础函数，其验证逻辑反转导致所有使用有效 hex 密钥的加密/解密操作全部失败。包括：
- 基础 hex 转换测试（3 个）
- 往返测试（2 个）
- 加密/解密功能测试（5 个）
- Unicode 边界测试（13 个）
- ASCII 特殊字符测试（9 个）
- 密钥相关测试（4 个）
- 性能和并发测试（部分）

**分析**：条件反转导致所有偶数长度的有效 hex 字符串被错误拒绝，而奇数长度的无效字符串反而被接受。由于 `hexToBytes` 是加密系统的核心依赖，此变异引发了大规模的级联失败，充分证明了测试体系对关键路径的覆盖。

## 改进建议

虽然本次审计结果良好（100% 杀死率），但仍有以下改进空间：

1. **扩大变异采样范围**：本次仅测试了 5 个模块的 6 个变异点，建议在未来审计中覆盖更多模块（如 services、store 层），以获得更全面的测试质量评估。

2. **引入自动化变异测试工具**：考虑在 CI 中集成 Stryker Mutator，实现持续的变异测试，而非一次性手动审计。自动化工具可以发现手动设计变异时可能忽略的盲区。

3. **M04 的测试敏感性**：M04 仅有 1 个测试用例捕获，建议增加更多边界测试（如 `"encrypt"`、`"encrypted"` 等以 `enc` 开头但无冒号的字符串），增强测试鲁棒性。

4. **级联依赖的风险**：M06 的 52 个失败测试中，大部分是级联效应而非直接测试 `hexToBytes` 的验证逻辑。建议增加直接针对 `hexToBytes` 奇偶验证的独立测试用例，使测试更具针对性。
